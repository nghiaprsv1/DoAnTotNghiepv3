1 Đọc & chuẩn hoá văn bản

Bước chuẩn bị để việc tách câu/chunk diễn ra chính xác trên mọi OS.

Đọc file nguồn

  • Thư mục: RAGV2_DOCS_DIR (env), mặc định ../documemtRAG tính từ process.cwd() của backend.
  • Chỉ nhận đuôi .txt / .md.
  • ingest(only?): truyền danh sách tên file → chỉ re-index các file đó (re-index từng phần).

4 bước chuẩn hoá (trong chunkText)

const text = raw.replace(/\r\n/g, '\n').trim();   // (1) chuẩn hoá xuống dòng + (2) cắt khoảng trắng thừa

// (3) tách theo đoạn — ranh giới là 1+ dòng trống
for (const block of text.split(/\n\s*\n+/)) {
  // (4) gộp dòng mềm (wrap) trong cùng đoạn thành một dòng dài
  const oneLine = block.replace(/\s*\n\s*/g, ' ').trim();
  if (oneLine) sentences.push(...splitToSentences(oneLine));
}
⚠️ Không bỏ dấu tiếng Việt ở khâu này Văn bản gốc giữ nguyên dấu để embedding nắm đúng ngữ nghĩa và hiển thị nguyên văn cho người dùng. Việc bỏ dấu câu / hạ chữ thường chỉ xảy ra ở khâu tokenize cho BM25 (mục 4).

2 Chunking theo câu trọn vẹn

File lib/chunker.ts — chiến lược sentence-aware + overlap theo câu.

Vì sao phải chunk?

Tài liệu dài 8–15 KB không thể embed nguyên khối: vector sẽ "loãng" và lúc truy hồi không định vị được đoạn liên quan. Chunk nhỏ → mỗi vector mang một ý cô đọng → truy hồi trúng hơn.

Tham số mặc định

Tham sốGiá trịÝ nghĩa
maxChars900Kích thước tối đa mỗi chunk (ký tự)
minChars350Đuôi ngắn hơn ngưỡng này sẽ gộp vào chunk trước (tránh chunk vụn)
overlapChars160Số ký tự gối đầu giữa 2 chunk (làm tròn LÊN theo câu trọn)

Điểm tinh tế: tách câu không cắt nhầm số

const SENTENCE_SPLIT = /(?<=[^\d\s][.!?…])\s+/u;

✓ "6. Di tích…"

Số thứ tự đầu mục — KHÔNG bị tách nhầm thành câu mới (ký tự trước dấu chấm là chữ số).

✓ "1.500.000" · "3,5 km"

Số tiền / số đo — KHÔNG bị cắt giữa số.

Minh hoạ overlap theo câu

Đoạn gốcCâu ACâu BCâu CCâu DCâu E
Chunk 0Câu ACâu BCâu C
Chunk 1Câu CCâu DCâu E

Câu C lặp lại ở cả 2 chunk = phần overlap, giữ mạch ngữ cảnh bắc cầu. Chunk sau luôn bắt đầu từ đầu một câu, không cắt giữa câu.

Các luật còn lại

  • Câu siêu dài (≥ maxChars): để câu đó đứng riêng thành một chunk — vẫn trọn câu.
  • Gộp đuôi ngắn: chunk cuối < minChars → nối vào chunk liền trước.
  • Kết quả mỗi chunk: { index, content, charCount }.

Tên gọi phương pháp

Hệ thống hiện dùng DUY NHẤT một phương pháp chunking:

Sentence-aware fixed-size chunking + sentence-level overlap

Tức là cắt theo kích thước cố định (~900 ký tự) nhưng tôn trọng ranh giới câu (không xẻ giữa câu) và gối đầu theo câu trọn (~160 ký tự). Chưa áp dụng semantic chunking (cắt theo độ tương đồng embedding) — đây thuần xử lý chuỗi, không tốn lần gọi LLM nào ở khâu chunk.

Khi nào chunking được chạy?

Chunking chỉ chạy ở khâu nạp tài liệu (ingest), OFFLINE — không chạy lúc người dùng hỏi. Cụ thể được gọi tại 2 thời điểm:

① Lệnh CLI

cd backend && npm run rag:ingest
→ rag-ingest.script.ts → RagV2Service.ingest() → gọi chunkText(raw) cho từng file.

② Nút trên trang

Bấm "Nạp tài liệu" ở /chatbot-v2 → POST /rag-v2/ingest → cùng ingest() đó.

💡 Chunking KHÔNG chạy lúc trả lời câu hỏi Khi người dùng hỏi, hệ thống chỉ embed câu hỏi rồi so cosine với các chunk đã được cắt & lưu sẵn trong bảng rag_knowledge_chunks. Câu hỏi người dùng đi theo lớp tiền xử lý query-time (tokenize/stopword ở mục 4), không qua chunker.
⚠️ Chỉ áp cho nhánh tài liệu Dữ liệu có cấu trúc (trips/places/guides/posts) KHÔNG đi qua chunking — query trực tiếp bảng DB (xem mục 5). Chunking chỉ dành cho tài liệu phi cấu trúc trong documemtRAG/.

3 Embedding & lưu trữ vector

Biến mỗi chunk thành vector số thực rồi lưu vào Postgres.

Sinh embedding

  • Provider trừu tượng RagEmbeddings — chọn qua env RAGV2_LLM_PROVIDER. Đổi provider không phải sửa pipeline ingest.
  • Model mặc định: gemini-embedding-001 (OpenAI: text-embedding-3-small).
  • taskType = RETRIEVAL_DOCUMENT khi index tài liệu (vs RETRIEVAL_QUERY khi embed câu hỏi) — gợi ý model tối ưu vector cho đúng vai trò.
  • Embed theo batch: gọi tuần tự từng chunk, có giãn cách RAGV2_EMBED_GAP_MS (1200ms) chống vượt rate limit free tier.
  • Retry + exponential backoff với 429/5xx: 2s → 4s → 8s → 16s → 32s (tối đa 5 lần).
⚠️ Số chiều embedding khác nhau giữa provider Gemini và OpenAI (1536) có số chiều vector khác nhau. Đổi RAGV2_LLM_PROVIDER bắt buộc chạy lại npm run rag:ingest, nếu không cosine sẽ lệch chiều.

Lưu trữ (vector store)

Entity KnowledgeChunk → bảng rag_knowledge_chunks:

Vì sao jsonb chứ không pgvector?

Không cần cài extension, chạy ngay trên Postgres hiện có (phù hợp quy mô đồ án). Đổi lại cosine tính brute-force trong Node, không có ANN index.

Re-index idempotent

Mỗi file: xoá toàn bộ chunk cũ cùng docName → save chunk mới (batch 50). Chạy lại nhiều lần không nhân bản.

4 Tiền xử lý tại thời điểm truy vấn

File lib/lexical.ts — áp lúc người dùng hỏi, dùng cho BM25 (sparse) và DB search.

Tokenize tiếng Việt

text.toLowerCase()
    .replace(/[^\p{L}\p{N}\s]/gu, ' ')   // bỏ dấu câu, giữ chữ + số mọi ngôn ngữ
    .split(/\s+/)
    .filter(t => t.length >= 2 && !STOPWORDS.has(t));
  1. Hạ chữ thường.
  2. Bỏ dấu câu / ký tự đặc biệt, giữ chữ và số (Unicode-aware \p{L}\p{N}).
  3. Tách theo khoảng trắng.
  4. Lọc token ngắn (< 2 ký tự) và stopword.
💡 Giữ nguyên dấu tiếng Việt (không bỏ dấu) "núi" ≠ "nui" → khớp chính xác hơn cho tài liệu tiếng Việt có dấu.

Stopword

Danh sách ~70 từ dừng tiếng Việt + Anh phổ biến (và, là, của, có, cho, các, một, được, the, a, is, to…). Loại bỏ để token mang nghĩa (địa danh, đặc sản, danh từ riêng) nổi lên, tăng tín hiệu cho BM25.

Áp dụng cho cả hai phía

Câu hỏi người dùng

Tokenize qua tokenize() trong bm25Scores().

Toàn bộ chunk

Tokenize thành LexicalDoc { id, tokens, length } để chấm BM25 — bổ trợ cho cosine (hybrid search).

5 Tiền xử lý nhánh dữ liệu có cấu trúc

Nguồn trips / places / guides / posts — không qua ingest/embedding.

Dữ liệu DB đã "sạch và có cấu trúc" sẵn, nên tiền xử lý chỉ là chuẩn hoá câu hỏi → token → khớp cột, diễn ra realtime mỗi truy vấn:

  • searchTokens() tách câu hỏi/keyword thành token để dựng điều kiện ILIKE + so token trên các cột text của entity.
  • Mỗi retriever lọc theo trạng thái hợp lệ: Trip status=published, Guide status=approved, Post visibility=public.
  • Chế độ browse: không có từ khoá lọc (ý định liệt kê chung) → bỏ điều kiện khớp text, trả top-N theo rating.
⚡ 0 token LLM ở bước này Toàn bộ là SQL + xử lý chuỗi thuần — nhanh, rẻ, không phụ thuộc quota LLM.

6 Cơ sở lý thuyết

Công thức các thuật toán/phương pháp dùng trong khâu tiền xử lý & truy hồi. Mọi công thức khớp đúng với code trong lib/.

6.1 · Embedding — biểu diễn vector dense

Hàm embedding ánh xạ một đoạn văn bản thành vector số thực ℝd sao cho hai đoạn gần nghĩa nằm gần nhau trong không gian vector:

E : text → ℝd,   v = E(chunk)
d = số chiều (Gemini gemini-embedding-001 · OpenAI text-embedding-3-small = 1536). Index tài liệu dùng taskType RETRIEVAL_DOCUMENT, embed câu hỏi dùng RETRIEVAL_QUERY.

Khi truy hồi, câu hỏi q cũng được embed thành E(q) rồi so độ tương đồng với mọi chunk đã lưu.

6.2 · Cosine similarity — truy hồi ngữ nghĩa dense

Đo góc giữa 2 vector: hướng giống nhau → tương đồng cao, không phụ thuộc độ dài vector. File lib/cosine.ts.

cos(a, b) = a · b ‖a‖  ‖b‖ = Σi ai bi √(Σ ai2) · √(Σ bi2)
Kết quả ∈ [−1, 1]. Trả 0 nếu một vector có độ dài 0 (tránh chia 0). Ngưỡng "grounded" trong hệ thống: cosine ≥ 0.3.
  • a · b = tích vô hướng (dot) · ‖a‖ = độ dài vector (magnitude)
  • Tính brute-force trên toàn bộ chunk (không pgvector/ANN) — phù hợp quy mô đồ án.

6.3 · BM25 — truy hồi từ khoá sparse

Chấm điểm khớp từ khoá giữa câu hỏi và chunk, bắt được tên riêng / mã số mà embedding hay "làm mượt" đi. File lib/lexical.ts.

score(q, d) = Σt∈q   IDF(t) · f(t,d) · (k1 + 1) f(t,d) + k1 · (1 − b + b · |d|/avgdl)
IDF(t) = ln( 1 + N − n(t) + 0.5 n(t) + 0.5 )
  • f(t,d) = tần suất term t trong chunk d  ·  |d| = số token của d  ·  avgdl = độ dài trung bình
  • N = tổng số chunk  ·  n(t) = số chunk chứa t (document frequency)
  • k₁ = 1.5 điều tiết bão hoà tần suất  ·  b = 0.75 điều tiết chuẩn hoá độ dài
  • Token hoá: hạ chữ thường, bỏ dấu câu (giữ dấu tiếng Việt), bỏ stopword + token < 2 ký tự.

6.4 · Hybrid Search + RRF — hợp nhất 2 bảng xếp hạng fusion

Dense (cosine ∈ [−1,1]) và sparse (BM25 ∈ [0,∞)) có thang điểm khác nhau → cộng thẳng là sai. Reciprocal Rank Fusion bỏ qua giá trị điểm, chỉ dùng thứ hạng. File lib/fusion.ts.

RRF(item) = ΣL 1 k + rankL(item)
k = 60 (theo Cormack 2009) — làm dịu ảnh hưởng của vài hạng đầu. Item xuất hiện hạng cao trong NHIỀU danh sách → điểm hợp nhất cao nhất.
  • L = mỗi danh sách xếp hạng (dense, sparse)  ·  rankL = thứ hạng 1-based của item trong L
  • Item không xuất hiện trong một danh sách → không cộng phần đó.

6.5 · Chunking — công thức gói câu

Phương pháp sentence-aware fixed-size + overlap (mục 2). Luật quyết định "chốt chunk":

chốt chunk khi  len(cur) + 1 + len(câu) > maxChars
maxChars = 900 · minChars = 350 (gộp đuôi vụn) · overlapChars = 160 (gối đầu theo câu trọn). Thuần đếm ký tự — không gọi LLM.
💡 Vì sao kết hợp dense + sparse? Dense (cosine) khớp theo ngữ nghĩa nhưng đôi khi bỏ sót từ khoá hiếm / tên riêng; sparse (BM25) khớp theo từ khoá chính xác. Hybrid + RRF lấy ưu điểm cả hai → recall tốt hơn hẳn dùng riêng một loại.

7 Vector Database (pgvector) THIẾT KẾ

Bản thiết kế nâng cấp vector store: từ jsonb + cosine brute-force → pgvector + HNSW index. Chi tiết đầy đủ ở docs/THIET-KE-VECTOR-DATABASE.md.

⚠️ Trạng thái: thiết kế, chưa triển khai code Hệ thống đang chạy lưu vector dạng jsonb (mục 3) và cosine brute-force trong Node. Phần dưới là blueprint để nâng cấp khi corpus lớn dần / để đo so sánh trong báo cáo.

7.1 · Vì sao nâng cấp?

Hiện cả 2 đường truy hồi đều this.chunks.find() — load toàn bộ chunk (kèm vector) vào Node rồi mới cosine. Đúng nhưng là O(N) và tốn băng thông DB↔Node.

pgvector — tại sao?

Extension trên chính Postgres sẵn có. Không phải dựng service riêng (Qdrant/Pinecone), ít hạ tầng nhất, hợp quy mô đồ án, nâng cấp tự nhiên từ code hiện tại.

HNSW index

Đồ thị "small-world" nhiều tầng → tìm láng giềng gần đúng ~O(log N) thay vì quét toàn bộ. Cosine qua toán tử <=>.

7.2 · Quyết định mấu chốt — số chiều embedding

Ràng buộc: gemini-embedding-001 trả 3072 chiều, nhưng HNSW của pgvector (kiểu vector) chỉ index tối đa 2000 chiều.

✓ Phương án A (đề xuất)

Giảm output về 1536 chiều: Gemini outputDimensionality=1536 (Matryoshka) + L2-normalize; OpenAI vốn đã 1536. → một schema vector(1536) chung cho 2 provider.

Phương án B

Giữ 3072 chiều, dùng kiểu halfvec(3072) (HNSW hỗ trợ tới 4000). Giữ trọn độ chính xác nhưng storage gấp đôi, lệch chiều với OpenAI.

7.3 · Schema thiết kế

-- (1) bật extension (Render PostgreSQL có sẵn pgvector) CREATE EXTENSION IF NOT EXISTS vector; -- (2) cột vector — giữ song song cột jsonb cũ để rollback / so sánh ALTER TABLE rag_knowledge_chunks ADD COLUMN IF NOT EXISTS embedding_vec vector(1536); -- (3) ANN index: HNSW + cosine CREATE INDEX IF NOT EXISTS idx_rag_chunks_hnsw ON rag_knowledge_chunks USING hnsw (embedding_vec vector_cosine_ops) WITH (m = 16, ef_construction = 64);

Cột vector + index quản lý bằng raw SQL trong ensureSchema() (TypeORM 0.3 không hiểu kiểu vector). Entity KnowledgeChunk giữ nguyên, không thêm @Column cho cột vector.

7.4 · Luồng truy hồi dense: trước → sau

HIỆN TẠI jsonb + brute-force

embed(câu hỏi) → vector
chunks.find() → load TOÀN BỘ vector vào Node
cosineSimilarity() từng chunk (JS)
sort → top-K · O(N)
→

THIẾT KẾ pgvector + HNSW

embed(câu hỏi) → vector
ORDER BY embedding_vec <=> $1 LIMIT k
Postgres tính cosine, dùng HNSW index
trả thẳng top-K · ~O(log N)
-- dense search có index (thay khối brute-force ở service) SELECT id, content, 1 - (embedding_vec <=> $1::vector) AS dense_score FROM rag_knowledge_chunks ORDER BY embedding_vec <=> $1::vector -- <=> = cosine distance LIMIT $2; -- denseK

Sparse (BM25) giữ ở Node nhưng chỉ SELECT id, content (không kéo cột vector) — khoản tiết kiệm băng thông lớn nhất. RRF (fusion.ts) giữ nguyên. Chỉ sửa 2 điểm brute-force: ragv2.service.ts:457 & :832.
